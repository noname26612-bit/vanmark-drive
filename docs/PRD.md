# PRD — Сервис задач водителей VanMark (рабочее имя: vanmark-drive)

Версия 1.0 · июнь 2026 · Владелец: Артём. Статус: согласовано для фазы 1 (MVP без 1С).

## 1. Проблема и цель

Сейчас задачи водителям ставятся в Telegram-чате «Доставки»: нумерованные сообщения, распределение вручную Миленой, отчёты — в переписке. Следствия: задачи теряются в ленте, нет статусов и истории, нет изоляции (все видят всё), прозрачность держится на памяти Милены.

Цель MVP: все задачи водителей живут в сервисе. Милена ставит и видит картину дня в реальном времени; водитель видит только свои задачи, ведёт статусы и сдаёт фото-отчёт. Ничего не теряется, история восстановима.

Метрики успеха пилота (2 недели): 100% новых задач заводятся в сервисе (не в чате); ноль «потерянных» задач; Милена отвечает на вопрос «где что находится» без звонка водителю.

## 2. Роли и пользователи

| Роль | Кто | Права |
|---|---|---|
| Диспетчер | Милена (гл. менеджер) | Создаёт/редактирует/назначает/отменяет задачи, видит всё, видит фото и историю; ведёт KPI водителей — подтверждает нарушения и закрывает месяц (см. §12) |
| Водитель | Алексей Каширский (Лада Ларгус, преимущественно выездные ремонты/диагностика); Алексей Писарев (преимущественно доставки) | Видит ТОЛЬКО свои задачи; меняет статусы; прикладывает фото/комментарии. Не создаёт и не редактирует задачи; видит свой расчёт зарплаты (только свой) |
| Админ | Артём | Всё + управление пользователями + денежные настройки KPI (оклад, премия, веса штрафов) |

Внешние исполнители (например, наёмный перевозчик Султан — «везёт Султан» в чате) заводятся как водители **без права входа**: задачи назначаются на них для полноты картины, статусы ведёт диспетчер.

Изоляция — жёсткое требование: водитель не может увидеть чужую задачу ни в списке, ни по прямой ссылке/ID (проверка на сервере, см. ARCHITECTURE §6).

## 3. Типы задач

Выведены из реального чата «Доставки»:

1. **Доставка в аренду** — станок + комплект (нож, размотчик, дог. маш.) на адрес клиента.
2. **Забор в ремонт** — забрать станок у клиента в ремонт/диагностику.
3. **Доставка/возврат из ремонта** — вернуть станок клиенту.
4. **Отвезти в ТК** — отгрузка в транспортную компанию (Энергия и др.), привязка к счёту.
5. **Забрать СДЭК/посылку** — забор по номеру накладной из ПВЗ/ТЦ.
6. **Выездной ремонт / диагностика** — работа мастера на объекте клиента. План может измениться по факту («скорее всего забираем к нам в цех») — диспетчер меняет тип, смена фиксируется в истории.
7. **Гарантийная замена** — привезти новый станок, забрать старый (одна поездка, два действия).
8. **Доставка проданного** — доставка нового купленного станка/комплекта покупателю.
9. **Закупка/выкуп станка** — забрать купленный станок (в т.ч. б/у у частника), возможна оплата на месте.
10. **Прочее** — свободный тип.

Тип влияет на: обязательность фото при завершении (см. §5), нужен ли подписанный акт (ремонтные типы — учитывается в KPI, см. §12), подсказки полей. Новые типы добавляются админом без правки кода (справочник).

## 5. Статусная модель

```
Новая → Назначена → Принята → В пути → На месте → Выполнена
              └────────── в любой момент: Перенесена (новая дата) / Отменена (причина) / Ждёт (причина: пропуск, запчасти)
```

Правила:
- Переходы по порядку; пропуск шага запрещён (нельзя из «Принята» сразу в «Выполнена»). Полная матрица переходов — ARCHITECTURE §5.
- Водитель двигает только вперёд по своей задаче + может написать комментарий. Диспетчер может корректировать любой статус (исправление ошибок), всё фиксируется в журнале.
- **«Выполнена» требует минимум 1 фото** для типов: доставка в аренду, забор в ремонт, возврат из ремонта, гарантийная замена, выездной ремонт. Для «Отвезти в ТК» и «Забрать СДЭК» — фото опционально (но поощряется: экран приёмки/накладная). Настройка обязательности — в справочнике типов.
- Каждая смена статуса автоматически пишет: кто, когда, гео-точку (если водитель дал разрешение на гео). Постоянного GPS-трекинга нет — только метки в моменты смены статуса.
- «Перенесена» возвращает задачу в «Назначена» на новую дату с сохранением истории.
- Если по задаче указана оплата «на месте» — завершение дополнительно требует отметки «Деньги получены» с суммой; отметка пишется в историю.
- Для ремонтных типов ожидается подписанный акт (документ-вложение). Его отсутствие, как и опоздание относительно окна времени, **не блокирует** завершение — это отметки KPI, а не запрет (в отличие от обязательного фото). Разбор и учёт — в §12.

## 6. Пропуска (важная специфика)

В чате это постоянная тема («заранее минимум за день скинуть данные для заказа пропуска»). Логика:
- У задачи статус пропуска: «не нужен» / «нужен, не заказан» / «заказан».
- Если на завтра есть задача со статусом «нужен, не заказан» — Милене показывается предупреждение в доске (блок «Требуют внимания») в течение дня накануне.
- Водитель видит статус пропуска в карточке (чтобы не уехать на объект без пропуска).

## 7. Уведомления

- Водителю (push на Android, PWA): новая задача назначена; задача изменена/перенесена/отменена; напоминание утром «у тебя N задач на сегодня» (08:00, настраиваемо).
- Милене (в интерфейсе, опционально push): водитель завершил задачу; водитель добавил комментарий/фото; задача «зависла» (в одном статусе дольше X часов — настройка, по умолчанию выключено в MVP, включаем после пилота).

## 8. Экраны

**Диспетчер (десктоп, браузер):**
1. «Сегодня» — главный экран: колонки по водителям (+ колонка «Не назначено»), внутри карточки задач со статус-цветом; переключатель вида «по статусам» (канбан). Счётчики сверху: всего/в работе/выполнено/требуют внимания. Живое обновление.
2. «Все задачи» — таблица с фильтрами (период, водитель, тип, статус, поиск по тексту/номеру/счёту).
3. Карточка задачи — все поля, лента истории (статусы с временем и гео), фото-галерея, комментарии, кнопки: редактировать, переназначить, перенести, отменить.
4. Создание задачи — одна быстрая форма, заполняется за ≤30 секунд; копирование задачи как шаблон.
5. «Водители» (админ): пользователи, активность; денежные настройки KPI — оклад/премия по водителю, веса штрафов, шаг прогрессии (см. §12).
6. «KPI / Зарплата» (диспетчер) — по каждому водителю за месяц: кандидаты в нарушения от системы (подтвердить/отклонить одним действием), добавить отметку или разовое поощрение вручную, текущий расчёт, кнопка «Закрыть месяц» (см. §12).

**Водитель (телефон, PWA, Android):**
1. «Мои задачи» — список на сегодня (по умолчанию), сортировка по времени; вкладка «Ближайшие» (завтра+). Карточки крупные: номер, тип-иконка, время, адрес, статус-цвет, индикатор пропуска. Чтобы назначенная задача не пропала из вида (у водителя нет экрана «все задачи»), вкладка «Сегодня» также показывает просроченные незавершённые и задачи без даты с пометками «Просрочено»/«Без даты» (решение Артёма, этап 3); завершённые за день сворачиваются вниз.
2. Карточка задачи — суть, адрес (кнопка «Навигатор»), контакт (кнопка «Позвонить»), счёт/оборудование, статус пропуска, блок оплаты крупно («Взять 5 000 ₽ на месте»); одна большая кнопка следующего статуса («Принял» → «Выехал» → «На месте» → «Выполнено»); фото + комментарий.
3. Экран завершения — обязательное фото (если тип требует), комментарий, подтверждение.
4. «История» — мои выполненные задачи (неделя/месяц).
5. «Мой расчёт» — оклад, премия, список моих нарушений за месяц (дата, № задачи, что нарушено, −сумма) и итог к выплате; переключатель месяца. Только просмотр, строго свои данные (см. §12).

UX-принципы: статус-цвета единые везде (серый — новая, фиолетовый — назначена, синий — принята/в пути, оранжевый — на месте, зелёный — выполнена, красный — отменена/проблема, жёлтый — ждёт); у водителя крупная типографика и тач-цели ≥48px; оптимистичный UI со скрытым ретраем при плохой сети; интерфейс целиком на русском.

## 9. Нефункциональные требования

- Скорость: открытие списка задач <1 с на 4G; действия — мгновенный отклик (оптимистичные обновления).
- Доступность сервиса: один VPS в РФ, даунтайм при деплое допустим в нерабочее время; бэкапы БД и фото ежедневно (см. ARCHITECTURE §8).
- Данные: хранение в РФ; HTTPS обязательно; пароли — хэши (argon2/bcrypt).
- Масштаб: до 10 водителей и 200 задач/день без переделок (запас ×5 от текущего).
- Объём фото: ~5–20 МБ/задача; сжатие на клиенте до ~1920px.

## 12. KPI и расчёт зарплаты (Фаза 1.5)

Зарплата водителя считается **от ошибок, а не от объёма работы** (решение Артёма, июнь 2026). Принцип: каждый месяц водитель стартует с полной премией, нарушения её уменьшают. Мало ошибок — премия остаётся (прибавка); много — премия уходит в минус и режет оклад (убавка). Одна механика даёт и прибавку, и убавку.

### 12.1 Метрики (нарушения)

1. **Опоздание** — приехал на объект (статус «На месте») позже окна времени задачи (`timeTo`).
2. **Не подписан акт на ремонте** — ремонтная задача завершена без приложенного подписанного документа. Касается **только ремонтных типов** (флаг `requiresSignedDoc` у типа, см. §3). У водителя-доставщика (Писарев) ремонтных задач нет → метрика к нему не применяется и в его расчёте не показывается. Разделение доставка/ремонт вытекает из типов задач, отдельно настраивать не нужно.
3. **Не выполнена точка** — назначенная на день задача не доведена до «Выполнено» к концу дня (без переноса/отмены диспетчером).

Опоздание и отсутствие акта **не блокируют** завершение задачи — это отметки KPI, а не запрет (в отличие от обязательного фото, которое блокирует, §5). Водителю важнее закрыть работу на объекте; разбор — позже.

### 12.2 Кто что делает

- **Система** — сама находит кандидатов в нарушения из данных задач (время статусов, наличие акта-документа, незавершённые задачи) и предлагает их Милене. Никто не вбивает рутину руками.
- **Милена (диспетчер)** — подтверждает или отклоняет каждого кандидата одним действием; может добавить нарушение или разовое поощрение вручную; в конце месяца «закрывает» период (фиксирует расчёт).
- **Артём (админ)** — задаёт денежные правила: оклад каждому, размер премии, вес каждого штрафа, шаг прогрессии.
- **Водитель** — видит свой расчёт в приложении (оклад, премия, список своих ошибок, итог). Только просмотр, строго свои данные (та же серверная изоляция, что и с задачами, — чужой расчёт по прямой ссылке отдаёт 404).

### 12.3 Формула (прогрессивная)

```
Итог = Оклад + (Премия − Штрафы) + Ручные поощрения
```

Штрафы прогрессивные: каждая следующая ошибка месяца дороже предыдущей (вес ошибки × коэффициент по её порядковому номеру в месяце). Премия может уйти в минус (убавка ниже оклада); итог к выплате не опускается ниже 0.

Пример (числа настраиваются админом):

| Ситуация за месяц | Расчёт | Итог |
|---|---|---|
| 0 ошибок | 50 000 + 10 000 | **60 000 ₽** (полная прибавка) |
| 2 опоздания | 50 000 + 10 000 − (1 000 + 1 250) | **57 750 ₽** |
| 3 опоздания + 1 без акта + 1 точка (на 13 000) | 50 000 + 10 000 − 13 000 | **47 000 ₽** (убавка) |

Дефолтные веса: опоздание 1 000 ₽, не выполнена точка 2 000 ₽, без акта 3 000 ₽; премия 10 000 ₽; прогрессия ×1.25 на каждую следующую ошибку. Оклад — по каждому водителю отдельно.

### 12.4 Что неизменяемо

- **Закрытие месяца** сохраняет расчёт снимком: позднее изменение правил/весов не переписывает уже выплаченное.
- Журнал задач (`TaskEvent`) остаётся неизменяемым (правило 3 CLAUDE.md). Отметки KPI — корректируемый реестр: пока месяц открыт, Милена может переотметить (с записью кто/когда); закрытый месяц неизменен.

### 12.5 Открытые вопросы по KPI

1. Точные суммы: оклад каждого водителя, размер премии, веса штрафов, шаг прогрессии — Артём задаёт перед запуском модуля (до этого — дефолты выше как заглушки).
2. Нижний порог итога: оставить 0 или ввести «не ниже X% оклада» — по умолчанию просто не ниже 0.
3. Поощрения (прибавки сверх премии): нужны ли отдельные положительные отметки за образцовую работу, или достаточно полной премии за чистый месяц — по умолчанию доступна ручная положительная отметка.
