# PRD — Сервис задач водителей VanMark (рабочее имя: vanmark-drive)

Версия 1.0 · июнь 2026 · Владелец: Артём. Статус: согласовано для фазы 1 (MVP без 1С).

## 1. Проблема и цель

Сейчас задачи водителям ставятся в Telegram-чате «Доставки»: нумерованные сообщения, распределение вручную Миленой, отчёты — в переписке. Следствия: задачи теряются в ленте, нет статусов и истории, нет изоляции (все видят всё), прозрачность держится на памяти Милены.

Цель MVP: все задачи водителей живут в сервисе. Милена ставит и видит картину дня в реальном времени; водитель видит только свои задачи, ведёт статусы и сдаёт фото-отчёт. Ничего не теряется, история восстановима.

Метрики успеха пилота (2 недели): 100% новых задач заводятся в сервисе (не в чате); ноль «потерянных» задач; Милена отвечает на вопрос «где что находится» без звонка водителю.

## 2. Роли и пользователи

| Роль | Кто | Права |
|---|---|---|
| Диспетчер | Милена (гл. менеджер) | Создаёт/редактирует/назначает/отменяет задачи, видит всё, видит фото и историю; **расценивает ведомости работ** (ставит цены, §13); ведёт KPI водителей — подтверждает нарушения и закрывает месяц (см. §12) |
| Водитель | Алексей Каширский (Лада Ларгус, преимущественно выездные ремонты/диагностика); Алексей Писарев (преимущественно доставки); Николай (штатный подменный — закрывает задачи водителей на больничном/в отпуске, любые типы) | Видит ТОЛЬКО свои задачи; меняет статусы; прикладывает фото/комментарии. Не создаёт и не редактирует задачи; видит свой расчёт зарплаты (только свой) |
| Админ | Артём | Всё + управление пользователями + денежные настройки KPI (оклад, премия, веса штрафов) |

Внешний перевозчик — наёмный (не штатный) исполнитель — заводится как водитель **без права входа**: задачи назначаются на него для полноты картины, статусы ведёт диспетчер. В интерфейсе отображается нейтрально — «Внешний перевозчик».

KPI/зарплата считаются только для водителей с заведённым денежным профилем (§12). Николай и внешний перевозчик в расчёт НЕ входят (профиля нет) — у Николая зарплата считается вне сервиса; экран «Мой расчёт» ему не показывается.

Изоляция — жёсткое требование: водитель не может увидеть чужую задачу ни в списке, ни по прямой ссылке/ID (проверка на сервере, см. ARCHITECTURE §6).

## 3. Типы задач

10 рабочих типов (порядок = очередь в списке у водителя и диспетчера) + служебный «Прочее». Перечень и очередность — решение Артёма (18.06.2026). Для каждого указано, нужен ли подписанный акт **по умолчанию** и нужна ли расценка работ (см. §13).

**С актом** (по умолчанию ожидается подписанный акт — мягко, влияет на KPI и бонус §12; диспетчер может снять требование галочкой на конкретной заявке, см. §4):
1. **Выездной ремонт / диагностика** — работа мастера на объекте клиента. Акт выполненных работ. **Требует расценки** (ведомость работ → Милена ставит цены, §13). План может измениться по факту («скорее всего забираем к нам в цех») — диспетчер меняет тип, смена фиксируется в истории.
2. **Забор в ремонт** — забрать станок у клиента в ремонт/диагностику. Акт приёма-передачи (опись оборудования). Без расценки.
3. **Гарантийный ремонт** — ремонт/замена по гарантии. Акт выполненных работ. По умолчанию бесплатно для клиента; при необходимости диспетчер указывает стоимость (тогда идёт через расценку, §13).
4. **Доставка в аренду** — станок + комплект (нож, размотчик, дог. маш.) на адрес клиента. Акт приёма-передачи (опись). Без расценки.
5. **Забор с аренды** — забрать арендованное оборудование у клиента. Акт возврата оборудования (опись/состояние). Без расценки.

**Без акта** (акт по умолчанию не нужен):
6. **Доставка из ремонта** — вернуть отремонтированный станок клиенту.
7. **Доставка проданного оборудования** — доставка нового купленного станка/комплекта покупателю.
8. **Сдача в ТК** — отгрузка в транспортную компанию (Энергия и др.), привязка к счёту.
9. **Забрать посылку** — забор по номеру накладной из ПВЗ/ТЦ (СДЭК и др.).
10. **Закупка/выкуп станка** — забрать купленный станок (в т.ч. б/у у частника), возможна оплата на месте.

**Прочее** — свободный служебный тип (без акта, без обязательного фото).

Тип задаёт значения по умолчанию: требуется ли акт, нужна ли расценка, подсказки полей. **Фото — всегда по желанию** на любом типе (не блокирует завершение и не влияет на KPI/бонус; решение 18.06.2026). Требование акта диспетчер может переопределить на конкретной заявке (§4). Новые типы добавляются админом без правки кода (справочник).

## 5. Статусная модель

```
[в списке: Новая → Назначена]  →  В работе  →  Завершена
   (у водителя статуса нет)          └── в любой момент: На паузе (причина) / Перенесена (дата) / Отменена (причина)
```

Переработка механики водителя (решение Артёма 19.06.2026). Правила:
- **До взятия в работу статуса у водителя нет** — задача просто в списке. «Новая»/«Назначена» — внутренние статусы для диспетчера (распределение).
- Рабочая цепочка **схлопнута**: одна кнопка «В работу» (прежние «Принята»+«В пути»+«На месте»), затем «Завершить». Время на задачу = «В работе» → «Завершена». Полная матрица — ARCHITECTURE §5.
- **Одна активная задача**: водитель берёт в работу только одну задачу; пока не завершил или не поставил на паузу — другую взять нельзя (проверка на сервере).
- «На паузе» (обязательная причина: пропуск, запчасти, клиента нет) **освобождает слот** — можно взять другую; вернуться к отложенной — снова «В работу».
- Водитель двигает только вперёд по своей задаче + может написать комментарий. Диспетчер может корректировать любой статус (исправление ошибок, ведение за внешнего перевозчика), всё фиксируется в журнале.
- **Фото — всегда по желанию** (решение 18.06.2026): не блокирует завершение ни на одном типе и не влияет на KPI/бонус. Прикладывается по необходимости (фото работы/оборудования как доп. подтверждение). Ранее фото было обязательным гейтом для ремонтных/арендных типов — это снято.
- Каждая смена статуса автоматически пишет: кто, когда, гео-точку (если водитель дал разрешение на гео). Постоянного GPS-трекинга нет — только метки в моменты смены статуса.
- «Перенесена» возвращает задачу в «Назначена» на новую дату с сохранением истории.
- Если по задаче указана оплата «на месте» — при завершении водитель **обязан отметить одно из двух**: «деньги получены» (с суммой) **либо** «не получены» с указанием причины (типовые: оплатят по счёту / оплата через офис / спор по сумме-работам / другое — свой комментарий). Завершить без оплаты можно (жёсткого запрета больше нет, решение Артёма 23.06.2026), но факт и причина сохраняются на задаче и неизменяемой записью в журнале; диспетчер видит заметную плашку «Не оплачено: причина». На KPI/зарплату неоплата НЕ влияет (информация для диспетчера, не штраф).
- Для типов с актом ожидается подписанный акт (документ-вложение — фото/скан бумажного бланка). Требование берётся из типа, но диспетчер может снять его на конкретной заявке (§4). Отсутствие требуемого акта, как и опоздание относительно окна времени, **не блокирует** завершение — это отметка KPI и недобор для бонуса (§12), а не запрет. Процесс формирования акта (ведомость работ → расценка → бумажный акт) — в §13.

### 5.1 Смена водителя (как у такси)

Рабочий день водителя оформляется сменой (решение Артёма 19.06.2026):
- Утром на базе водитель нажимает **«Открыть смену»** — это фактическое время начала работы. Открыть нужно до **9:15** (9:00 + 15 мин запаса), иначе — отметка KPI «поздно открыл смену» (§12.1).
- Открытие **подтверждает диспетчер** (Милена на базе и видит приход физически). До подтверждения смена «ждёт подтверждения»; зачётное время начала — момент нажатия водителя.
- Вечером водитель нажимает **«Закрыть смену»** — конец рабочего дня.
- **Брать задачу в работу можно только при открытой смене** (решение Артёма 19.06.2026): без открытой смены приложение не даёт начать задачу (сервер отклоняет — `SHIFT_REQUIRED`). Подтверждения Милены при этом ждать не нужно — достаточно открыть смену.
- По смене считается: **отработано** (сумма времени по задачам «В работе → Завершена») и **простой** (длина смены − отработано). Видно в «Сводке» у диспетчера. Простой — пока только показатель, без штрафа.
- Подменный водитель (Николай) и внешний перевозчик смену под KPI не ведут (нет денежного профиля). Пороги (начало дня 9:00 / запас 15 мин / рабочий день) — настройки админа.

## 6. Пропуска (важная специфика)

В чате это постоянная тема («заранее минимум за день скинуть данные для заказа пропуска»). Логика:
- У задачи статус пропуска: «не нужен» / «нужен, не заказан» / «заказан».
- Если на завтра есть задача со статусом «нужен, не заказан» — Милене показывается предупреждение в доске (блок «Требуют внимания») в течение дня накануне.
- Водитель видит статус пропуска в карточке (чтобы не уехать на объект без пропуска).

## 7. Уведомления

- Водителю (push на Android, PWA): новая задача назначена; задача изменена/перенесена/отменена; напоминание утром «у тебя N задач на сегодня» (08:00, настраиваемо).
- Милене (в интерфейсе, опционально push): водитель завершил задачу; водитель добавил комментарий/фото; задача «зависла» (в одном статусе дольше X часов — настройка, по умолчанию выключено в MVP, включаем после пилота).

## 8. Экраны

**Диспетчер (десктоп, браузер):**
1. «Сегодня» — главный экран: колонки по водителям (задачи на сегодня) + пул «Без даты» + пул «Ближайшие 3 дня» (сегодня…+2 дня для планирования наперёд: сегодня — только нераспределённые, чтобы не дублировать колонки водителей; завтра/послезавтра — все задачи). Внутри карточки задач со статус-цветом; перетаскивание для назначения/переноса в «Без даты». Счётчики сверху: всего/в работе/выполнено/требуют внимания/не назначено сегодня. Живое обновление. (Пул «Не назначено» убран — его роль закрывает «Ближайшие 3 дня».) Пулы (служебные и колонки водителей) можно менять местами перетаскиванием за шапку и сворачивать в узкую полосу (свёрнутый освобождает место — остальные расширяются), чтобы прятать второстепенные (внешний перевозчик, подменный). Порядок и свёрнутость запоминаются в аккаунте (на любом устройстве). Сверху — блок «Открытие смен»: запросы водителей на открытие смены с кнопкой «Подтвердить» (§5.1).
2. «Планирование» — сетка «водители × дни» на неделю вперёд (скользящее окно от сегодня; ‹ › листают по неделям). Строки: «Без водителя» (задачи с датой без исполнителя) + по строке на водителя; столбцы — дни; снизу пул «Без даты». Перетаскивание карточки в ячейку задаёт день и водителя одним действием; в «Без даты» — снимает дату. Строки-пулы можно менять местами перетаскиванием за левую метку (порядок запоминается в аккаунте). Назначение для распределения нагрузки наперёд (в отличие от «Сегодня» — исполнения текущего дня).
3. «Все задачи» — таблица с фильтрами (период, водитель, тип, статус, поиск по тексту/номеру/счёту).
4. Карточка задачи — все поля, лента истории (статусы с временем и гео), фото-галерея, документы (акты), комментарии, кнопки: редактировать, переназначить, перенести, отменить; расценка ведомости работ (проставить цены по позициям, §13); признак комплектности акта.
5. Создание задачи — одна быстрая форма, заполняется за ≤30 секунд; копирование задачи как шаблон.
6. «Водители» (админ): пользователи, активность; денежные настройки KPI — оклад/премия по водителю, веса штрафов, шаг прогрессии (см. §12).
6. «KPI / Зарплата» (диспетчер) — по каждому водителю за месяц: кандидаты в нарушения от системы (подтвердить/отклонить одним действием), добавить отметку или разовое поощрение вручную, текущий расчёт, кнопка «Закрыть месяц» (см. §12).
7. «Сводка» (диспетчер/админ, Фаза 2) — управленческая картина по водителям за период (день/неделя/месяц, по дате закрытия задач): выполнено задач и разбивка по типам (ремонты/доставки), опоздания и невыполненные точки (из подтверждённых отметок KPI), отмены и переносы, среднее время на объекте; выгрузка в CSV. Только просмотр; это операционная сводка, а не KPI-расчёт и не телематика/AI.
8. «Календарь загрузки» (диспетчер/админ, Фаза 2) — сетка «водители × дни» на 2 недели: загрузка дня в часах (от рабочего дня) и числе задач со статус-цветом, клик по ячейке → задачи дня; подсказка «кто свободен» с учётом специализации водителя. В навигации — **после «Планирования»**. Только просмотр (планирование/назначение остаются на «Сегодня»/«Планировании»). Подробно — §14.

**Водитель (телефон, PWA, Android):**
1. «Мои задачи» — вверху **блок смены** (открыть/ждёт подтверждения/закрыть, §5.1); список на сегодня (по умолчанию), сортировка по времени; вкладка «Ближайшие» (завтра+). Карточки крупные: номер, тип-иконка, время, адрес, статус-цвет, индикатор пропуска. Чтобы назначенная задача не пропала из вида (у водителя нет экрана «все задачи»), вкладка «Сегодня» также показывает просроченные незавершённые и задачи без даты с пометками «Просрочено»/«Без даты» (решение Артёма, этап 3); завершённые за день сворачиваются вниз.
2. Карточка задачи — суть, адрес (кнопка «Навигатор»), контакт (кнопка «Позвонить»), счёт/оборудование, статус пропуска, блок оплаты крупно («Взять 5 000 ₽ на месте»); одна большая кнопка следующего статуса («В работу» → «Завершить»; из паузы — «Вернуть в работу»), заблокирована, если уже есть активная задача (§5); фото + комментарий; для типов с расценкой — блок «Ведомость работ» (выбор выполненных работ без цен, «Отправить на расценку», §13); прикрепление фото подписанного акта.
3. Экран завершения — обязательное фото (если тип требует), комментарий, подтверждение.
4. «История» — мои выполненные задачи (неделя/месяц).
5. «Мой расчёт» — оклад, премия, список моих нарушений за месяц (дата, № задачи, что нарушено, −сумма) и итог к выплате; прогресс бонуса за комплектность актов (§12.6); переключатель месяца. Только просмотр, строго свои данные (см. §12).

## 12. KPI и расчёт зарплаты (Фаза 1.5)

Зарплата водителя считается **от ошибок, а не от объёма работы** (решение Артёма, июнь 2026). Принцип: каждый месяц водитель стартует с полной премией, нарушения её уменьшают. Мало ошибок — премия остаётся (прибавка); много — премия уходит в минус и режет оклад (убавка). Одна механика даёт и прибавку, и убавку.

### 12.1 Метрики (нарушения)

1. **Поздно открыл смену** — фактическое открытие смены (§5.1) позже порога 9:15 (начало дня + запас). Заменяет прежнее «опоздание на объект» (решение Артёма 19.06.2026): контроль перешёл с приезда на конкретную точку на начало рабочего дня.
2. **Не подписан акт** — задача, по которой акт требуется (по типу или по флагу заявки, §3–§4), завершена без приложенного подписанного документа. Применяется к **обоим водителям**: у Писарева тоже есть актовые задачи (доставка в аренду, забор с аренды). Если диспетчер снял требование акта на заявке — метрика к ней не применяется. (Прежнее ограничение «только ремонтные типы / не для Писарева» отменено — решение 18.06.2026.)
3. **Не выполнена точка** — назначенная на день задача не доведена до «Выполнено» к концу дня (без переноса/отмены диспетчером).

Опоздание и отсутствие акта **не блокируют** завершение задачи — это отметки KPI, а не запрет (в отличие от обязательного фото, которое блокирует, §5). Водителю важнее закрыть работу на объекте; разбор — позже.

### 12.2 Кто что делает

- **Система** — сама находит кандидатов в нарушения из данных задач (время статусов, наличие акта-документа, незавершённые задачи) и предлагает их Милене. Никто не вбивает рутину руками.
- **Милена (диспетчер)** — подтверждает или отклоняет каждого кандидата одним действием; может добавить нарушение или разовое поощрение вручную; в конце месяца «закрывает» период (фиксирует расчёт).
- **Артём (админ)** — задаёт денежные правила: оклад каждому, размер премии, вес каждого штрафа, шаг прогрессии.
- **Водитель** — видит свой расчёт в приложении (оклад, премия, список своих ошибок, итог). Только просмотр, строго свои данные (та же серверная изоляция, что и с задачами, — чужой расчёт по прямой ссылке отдаёт 404).

### 12.3 Формула (прогрессивная)

```
Итог = Оклад + (Премия − Штрафы) + Ручные поощрения
```

Штрафы прогрессивные: каждая следующая ошибка месяца дороже предыдущей (вес × коэффициент прогрессии по порядковому номеру ошибки в месяце; прогрессия включается начиная с заданного по счёту нарушения). Нижний порог итога настраивается (PayoutFloor): по решению Артёма (17.06.2026) — **не ниже оклада** (штрафы максимум обнуляют премию, оклад не режется); альтернативный режим — «не ниже 0» (премия уходит в минус и режет оклад). Ручные поощрения добавляются сверх.

Пример (числа Артёма, зафиксированы 17.06.2026: оклад 70 000, премия 40 000; веса — опоздание 500, невыполненная точка 500, без акта 1 000; прогрессия ×1,1 начиная с 3-го нарушения; порог — не ниже оклада):

| Ситуация за месяц | Расчёт | Итог |
|---|---|---|
| 0 ошибок | 70 000 + 40 000 | **110 000 ₽** (полная премия) |
| 2 опоздания | 70 000 + (40 000 − (500 + 500)) | **109 000 ₽** |
| 3 опоздания + 1 без акта + 1 точка | 70 000 + (40 000 − (500+500+550+1 210+666)) | **106 574 ₽** |
| штрафов больше премии | 70 000 + max(0, премия − штрафы) | **70 000 ₽** (не ниже оклада) |

> Историческая проверка формулы (старые дефолты: оклад 50 000, премия 10 000, веса 1 000/2 000/3 000, прогрессия ×1.25 с каждой следующей) даёт «2 опоздания → 57 750 ₽» — этот случай закреплён в unit-тестах `src/domain/kpi.test.ts`.

Технически: веса штрафов — глобальные (`KpiRule`); оклад/премия — на водителя (`DriverPayProfile`); шаг прогрессии, порядок её включения и нижний порог — глобальные (`KpiSettings`). Всё настраивается админом в UI (экран «Оплата (KPI)»).

### 12.4 Что неизменяемо

- **Закрытие месяца** сохраняет расчёт снимком: позднее изменение правил/весов не переписывает уже выплаченное.
- Журнал задач (`TaskEvent`) остаётся неизменяемым (правило 3 CLAUDE.md). Отметки KPI — корректируемый реестр: пока месяц открыт, Милена может переотметить (с записью кто/когда); закрытый месяц неизменен.

### 12.6 Бонус за комплектность документов (решение Артёма, 18.06.2026)

Сверх премии — фиксированный **бонус 5 000 ₽** за дисциплину по актам. Считается **помесячно**, по каждому водителю:

- **База (знаменатель)** — завершённые за месяц задачи, по которым акт **фактически требуется** (с учётом галочек-исключений §4).
- **Комплект (числитель)** — те из них, где приложен подписанный акт (документ-вложение). Только акт; фото на бонус не влияет.
- Если `комплект / база ≥ 80%` → **+5 000 ₽** к выплате.
- База = 0 (за месяц не было актовых задач) → бонус не начисляется (нейтрально, без штрафа).
- Применяется к обоим водителям одинаково (у Писарева актовые задачи — доставка в аренду / забор с аренды).

Бонус виден и водителю («Мой расчёт»), и диспетчеру («KPI / Зарплата») с прогрессом: например «Акты: 17/20 = 85% → бонус 5 000 ₽ начислен» или «72% — не хватает 2 актов до бонуса». Технически — отдельный компонент расчёта поверх формулы §12.3 (не штраф и не ручное поощрение); порог 80% и сумма 5 000 ₽ — настройка в `KpiSettings`.

## 13. Ведомость работ, расценка и акт (Фаза 2)

Заменяет нынешний телефонный процесс (водитель звонит → менеджер диктует цены → водитель пишет от руки → подписывают), сохраняя правило: **водитель фиксирует факт работ, но не формирует цены** (нет доступа к деньгам). Решения Артёма 18.06.2026.

### 13.1 Процесс (выездной ремонт / диагностика, платная гарантия)

1. Водитель на объекте, ремонт сделан (статус «На месте»). В карточке задачи заполняет **ведомость работ**: выбирает выполненные работы из справочника (+ свободные строки) и количество. **Без цен.**
2. «Отправить на расценку» → ведомость переходит в состояние «На расценке», Милене приходит сигнал «водитель ждёт цен» (пуш + приоритет на доске).
3. Милена проставляет цены по позициям (справочник работ как подсказка, цену правит под случай — прайс частичный) и подтверждает. Ведомость → «Расценён».
4. Водитель видит готовые суммы на экране, **переносит их в бумажный бланк акта**, заказчик подписывает на месте. Электронной подписи на экране пока нет — отложено (§10), акт бумажный.
5. Водитель фотографирует подписанный акт → прикладывает (документ-вложение) → завершает задачу. Ведомость → «Подписан».

Процесс **синхронный**: подпись возможна только на объекте в моменте, поэтому водитель ждёт расценку на месте; расценка должна быть быстрой. Если расценку в моменте не дали — задача всё равно закрывается, фото акта прикладывается позже, отсутствие требуемого акта учитывается в §12.

### 13.2 Акты-описи (забор в ремонт, доставка в аренду, забор с аренды)

Без расценки. Это **опись оборудования** (что приняли/выдали/вернули, состояние, комплектность) + подпись заказчика. Бумажный бланк, фото подписанного → документ-вложение. Ведомость цен не заполняется.

### 13.3 Справочник работ

Список типовых работ и запчастей для выездного ремонта (ведём как справочник, аналогично типам задач). Частично с ценами-подсказками; Милена корректирует под конкретный случай. Пополняется админом без правки кода.

### 13.4 Жизненный цикл акта (не меняет статусную модель задачи §5)

```
Черновик (водитель заполняет) → На расценке (ждёт Милену)
   → Расценён (цены проставлены) → Подписан (фото бумажного акта приложено)
```

Состояния акта живут отдельно от статуса задачи; завершение задачи (DONE) не блокируется состоянием акта. Назначение акта — доказательное (на случай спора об оплате), не бухгалтерская первичка; поэтому достаточно подписанного бумажного бланка с фото-копией в системе.

## 14. Ёмкость задачи и календарь загрузки (Фаза 2)

Подход осознанно лёгкий: расстояние считаем по прямой с поправочными коэффициентами; настоящий маршрут по дорогам с живыми пробками — отложен (§14.6). Модель данных — ARCHITECTURE §4б, этапы — ROADMAP (Фаза 2, Этапы 16–17).

### 14.1 Оценка времени задачи

У каждой задачи появляется **оценка времени** — сколько она займёт целиком (работа на объекте + дорога). Считается автоматически при создании; диспетчер может перебить вручную (тогда система не пересчитывает). Не задал — остаётся расчёт системы. При изменении адреса/даты/времени/типа оценка пересчитывается, если не задана вручную.

Формула:
```
Оценка (мин) = норма работы по типу + время в пути
время в пути  = (расстояние по прямой база→точка × коэфф. петляния)
                ÷ средняя скорость × коэфф. пробок (по времени выезда)
```
- **Норма работы** — фиксированное время «на объекте» по типу задачи (§14.2).
- **Расстояние** — по прямой от базы до адреса задачи. Адрес геокодируется в координаты один раз при создании (внешний геокодер — единственная внешняя зависимость; ARCHITECTURE §4б). Если адрес не распознан — оценка = только норма работы, с пометкой «дорога не учтена».
- **Коэфф. петляния** (дорога не по прямой) — 1.1 (настройка).
- **Средняя скорость** свободной дороги — настройка; 50 км/ч (подтверждено Артёмом 19.06.2026), калибруется по факту.
- **Коэфф. пробок** — множитель по времени суток (§14.3), берётся по началу окна задачи (`timeFrom`); время не задано → дневной коэффициент.
- Дорога считается **в одну сторону** (база→точка) как доля дороги на задачу; учёт обратной дороги — опциональная настройка, по умолчанию выключена.

**База компании:** ул. Ленина, 1Ас26, пос. Лесные Поляны, г.о. Пушкинский, Московская обл., 141212 (координаты 55.959611, 37.864076).

### 14.4 Календарь загрузки (новая вкладка, после «Планирования»)

Диспетчерский экран (только диспетчер/админ, изоляция как у «Сводки»): сетка **водители × дни** на 2 недели вперёд. В каждой ячейке — загрузка дня: **`6 ч 30 мин из 8 ч · 5 задач`** + статус-цвет (зелёный/жёлтый/красный) по проценту от рабочего дня. Клик по ячейке → список задач этого дня.

- **Рабочий день** — 9:00–18:00 минус 1 час обед = 8 рабочих часов (настройка). Водители иногда выезжают раньше (напр. на металлобазу к 4:00) или работают позже — в этом случае день просто покажет загрузку выше 100% (красным); система это не запрещает.
- **Поведение — только подсказка** (решение Артёма 19.06.2026): ничего не блокирует и не запрещает (в духе проекта — как KPI-отметки). Перегруз подсвечивается цветом.
- В календаре показываются **оба водителя + внешний перевозчик Султан** (Артём: включать).
- Рабочие дни — пн–пт; сб/вс показываются, но помечены как нерабочие (сб «по загруженности», вс — очень редко).
- Бонусом маленький индикатор загрузки можно показать и на ячейках вкладки «Планирование».

### 14.5 Подсказка «кто свободен» со специализацией

При выборе водителя для задачи календарь мягко рекомендует подходящего: учитывает и текущую загрузку, и специализацию (Артём: учитывать). Каширский — преимущественно выездные ремонты; Писарев и Султан — доставки. Это **подсказка, не запрет** — диспетчер может назначить любого. «Ремонтность» задачи определяется по признаку «требует расценки» (выездной ремонт + гарантийный ремонт); специализация водителя — отдельное поле в профиле.
