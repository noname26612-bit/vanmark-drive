# ROADMAP — vanmark-drive

Версия 1.0 · июнь 2026. Порядок работ для реализации в Claude Code. Один этап = одна рабочая сессия-две, одна ветка, свой критерий приёмки. Не перепрыгивать этапы; Definition of Done каждого — в CLAUDE.md.

## Фаза 1 — MVP (цель: команда работает в сервисе вместо Telegram-чата)

**Этап 0. Каркас проекта** — ~полдня
Репозиторий, Next.js (App Router, TS strict) + Prisma + PostgreSQL в docker compose для разработки, lint/typecheck/Vitest/Playwright подключены, `/api/health`, CLAUDE.md и skills скопированы в репо, docs/ заполнен этим пакетом.
✅ Приёмка: `pnpm dev` поднимает пустое приложение, все проверки зелёные, тестовый e2e проходит.

**Этап 1. Пользователи и вход** — ~полдня
Auth.js (логин+пароль, argon2id, JWT-cookie), роли ADMIN/DISPATCHER/DRIVER, сид: Артём (админ), Милена (диспетчер), водители Алексей Каширский и Алексей Писарев. Rate limit на вход. Заглушки экранов по ролям с редиректами.
✅ Приёмка: каждый из четырёх входит и попадает на свой экран; неверный пароль ×10 — блокируется; e2e на вход.

**Этап 2. Задачи: модель и работа диспетчера** — 1–2 дня
Схема Task/TaskType/TaskEvent (ARCHITECTURE §4), статусная матрица в `src/domain/`, сквозная нумерация (старт — последний номер из чата, уточнить у Артёма). Экраны Милены: «Сегодня» (колонки по водителям + «Не назначено», пул «Без даты», dnd-назначение), «Все задачи» (фильтры/поиск), карточка задачи с историей, форма создания ≤30 сек. Справочник типов в админке.
✅ Приёмка: Милена создаёт задачу всех 10 типов, назначает перетаскиванием, переносит/отменяет с причиной; история пишется; unit-тесты матрицы зелёные.

**Этап 3. PWA водителя и изоляция** — 1–2 дня
Экраны водителя: «Мои задачи» (сегодня/ближайшие), карточка (Навигатор, Позвонить, индикатор пропуска), кнопка следующего статуса с гео-меткой, комментарии. Изоляция по ARCHITECTURE §6. Оптимистичный UI с ретраем.
✅ Приёмка: полный прогон security-check (все тесты изоляции зелёные); водитель проходит цепочку статусов с телефона; чужие задачи недоступны ни списком, ни по id.

**Этап 4. Фото-отчёты** — ~1 день
Загрузка фото с камеры (сжатие до ~1920px), привязка к задаче, серверное требование фото для DONE по типу, галерея в карточке у Милены, раздача файлов с проверкой прав.
✅ Приёмка: DONE без фото для «Доставки в аренду» невозможен; фото видно Милене и не видно второму водителю; security-check на эндпоинты файлов.

**Этап 5. Push и PWA-установка** — ~1 день
Web App Manifest, Serwist, установка на домашний экран Android; web-push (VAPID): новая задача / изменение / отмена; утреннее напоминание (08:00) и предупреждение Милене о незаказанных пропусках на завтра (16:00) через node-cron.
✅ Приёмка: на реальном Android-телефоне: PWA установлена, пуш о новой задаче приходит при закрытом приложении, тап открывает карточку.

**Этап 6. Живые обновления и полировка** — ~1 день
SWR-поллинг 10 с без дёрганья раскладки, счётчики и блок «Требуют внимания» на «Сегодня», пустые/ошибочные состояния по ui-guidelines, прогон всех экранов на 360×740.
✅ Приёмка: изменение от водителя видно у Милены ≤10 с без F5; чек-лист ui-guidelines пройден по всем экранам.

**Этап 7. Прод и запуск** — ~1 день
VPS в РФ, Docker Compose + Caddy (HTTPS), домен, бэкапы с проверкой восстановления, сид прода, прогон deploy-release целиком, smoke на проде.
✅ Приёмка: сервис доступен по https, вход с телефонов водителей, бэкап восстановлен на тестовом контейнере.

**Этап 8. Пилот 2 недели** — финал фазы 1
Все новые задачи — в сервисе (чат — только для общения). Сбор фидбека от Милены и водителей, быстрые правки. Метрики из PRD §1.
✅ Приёмка фазы 1: Артём подтверждает — чат больше не используется для постановки задач.

Оценка фазы 1: ~6–9 рабочих дней чистой работы в Claude Code.

## Фаза 2 — по итогам эксплуатации (приоритизирует Артём)

Берётся в работу после приёмки фазы 1, по приоритету Артёма:

Карта на «Сегодня» (MapLibre/OSM, бесплатно для диспетчеризации); аналитика и отчёты (задачи/время по водителям, выгрузка в Excel); подпись клиента и чек-листы выездного ремонта; оптимизация порядка точек дня; S3 для фото; Telegram-дубль уведомлений; SSE вместо поллинга — только если реально нужно.

Интеграция с 1С:УНФ снята из плана (продуктовое решение Артёма от 04.06.2026) — сервис работает полностью автономно. Наработки сохранены как задел в `docs/archive/1C-INTEGRATION.md` (режимы обмена, маппинг, надёжность) и скилл `docs/archive/skills/1c-sync/`; поле `Task.externalId1C` из схемы убрано. Если интеграцию решат вернуть — это отдельная фаза: восстановить документ и скилл из архива, добавить `externalId1C` аддитивной миграцией. До явного решения Артёма тему не трогаем.

## Риски

| Риск | Митигация |
|---|---|
| Команда продолжит ставить задачи в чате по привычке | Пилот с жёстким правилом «задача = сервис», Милена — главный союзник; форма создания ≤30 сек |
| Фото съедят диск VPS | Сжатие на клиенте, мониторинг места в админке, S3 в фазе 2 |
| Слабый интернет у водителя на объекте | Оптимистичный UI + ретраи; критические действия не теряются |
| Расползание скоупа («давай ещё CRM/склад/маршруты») | Roadmap — закон; новые идеи — в бэклог фазы 2 |
