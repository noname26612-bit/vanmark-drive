# Интеграция с 1С — рабочие правила (отложено)

Перед любой работой прочитай `docs/archive/1C-INTEGRATION.md` целиком. Не начинать до явного решения Артёма вернуть интеграцию и ответа партнёра-хостера (режим A — прямой доступ, или режим B — 1С-инициатор).

## Принципы

1. **1С — мастер по данным заказа** (контрагент, адрес, счёт, дата); **сервис — мастер по исполнению** (статусы, водитель, фото, события). Перезапись только в этих направлениях.
2. **Идемпотентность везде.** Ключ объекта — `Ref_Key` (GUID 1С) в `task.externalId1C`: повторная загрузка = upsert. Подтверждения в 1С несут UUID операции — повтор не создаёт дубль.
3. **Ничего не терять.** Исходящие подтверждения — через таблицу `OutboxEvent` с ретраями (экспоненциальная задержка, без потолка попыток). Обрыв связи на сутки — норма, очередь догоняет.
4. **Журналировать всё** в `SyncLog` (направление, объект, payload-сводка, результат, текст ошибки 1С). Ошибки 1С часто невнятные — сохраняй сырой ответ.
5. Весь код в `src/integrations/onec/`: `client.ts` (транспорт), `mapping.ts` (поля 1С ↔ Task), `sync.ts` (логика), независимо от режима A/B.

## Особенности OData 1С (режим A)

- 1С отдаёт **OData v3** (не v4): не тащить тяжёлые SDK под v4 — обычный fetch + ручные запросы надёжнее.
- Чтение инкрементально: `$filter` по полю даты изменения, `$select` только нужных полей, `$top` + постраничность. Не выкачивать справочники целиком в цикле.
- Маппинг по `Ref_Key`; человекочитаемый номер документа — отдельно (`Number`), для показа Милене.
- **Запись в 1С через OData не делать** (табличные части передаются целиком — хрупко). Запись только через HTTP-сервис 1С (`POST /done`).
- Auth: basic поверх HTTPS, сервисный пользователь, креды в `.env` (`ONEC_BASE_URL`, `ONEC_USER`, `ONEC_PASSWORD`).
- Расписание: поллинг заказов каждые 2 мин (node-cron); выгрузка outbox — сразу при событии + догоняющий проход каждые 5 мин.

## Режим B (1С-инициатор)

- Эндпоинты `/api/1c/*` аутентифицируются по `X-Api-Key` (длинный случайный ключ в `.env`, отдельный от пользовательских сессий) + допустимо ограничение по IP.
- `GET /api/1c/outbox?after=<cursor>` отдаёт события пачками с курсором; 1С подтверждает получение (`POST /api/1c/ack`), только после ack события помечаются доставленными.
- Расширение для 1С (регл. задание + обработчики) пишем в отдельном репозитории/папке `onec-extension/`, тестируем ТОЛЬКО на копии базы, никогда на рабочей.

## Отладка и тесты

- Мок-сервер 1С для разработки: фикстуры реальных JSON-ответов OData (попросить выгрузку у Артёма/партнёра) в `src/integrations/onec/__fixtures__/`.
- Тесты: upsert-идемпотентность (двойная загрузка одного Ref_Key = одна задача), ретраи outbox, конфликт правок (поля 1С перезаписывают заказные поля, не трогая операционные).
- Админ-экран «Синхронизация»: статус последнего обмена, размер очереди, последние ошибки — делается в первом же этапе фазы 2, без него отлаживать вслепую.

## Запрещено

- Менять типовую конфигурацию УНФ (только расширение). Писать в 1С напрямую через OData. Хранить креды 1С вне `.env`. Запускать обмен на рабочей базе без согласования с Артёмом.
