# 1C-INTEGRATION — обмен с 1С:УНФ (отложено, бывшая фаза 2)

Версия 1.0 · июнь 2026. Контекст: 1С:УНФ, клиент-серверная база (SQL) в аренде/хостинге у партнёра, вход тонким клиентом. Сервис vanmark-drive к моменту интеграции уже работает автономно (фаза 1).

## 1. Цель обмена

1С остаётся источником правды по клиентам, заказам и деньгам. Сервис — операционный слой «кто куда едет и что сделал». Обмен убирает двойной ввод:

- **1С → сервис:** заказы/заявки, из которых рождаются задачи (контрагент, контакт, адрес, состав, счёт), изменения и отмены.
- **Сервис → 1С:** факт выполнения (статус, время прибытия/завершения, комментарий, ссылка на фото-отчёт), чтобы менеджеры видели исполнение прямо в 1С.

Важно: задачи можно создавать и руками в сервисе (как в фазе 1) — интеграция дополняет, а не заменяет ручной поток. Не каждая задача обязана иметь документ в 1С.

## 2. Два режима — выбор зависит от ответа партнёра

### Режим A (предпочтительный): прямой доступ — партнёр публикует базу

Партнёр публикует базу на веб-сервере (Apache/IIS) с HTTPS. Используем два штатных механизма:

- **OData `/odata/standard.odata` — чтение.** Сервис читает контрагентов, заказы, контакты с фильтром по дате изменения (инкрементальный поллинг каждые 1–5 минут). Состав объектов OData включается на стороне 1С (`УстановитьСоставСтандартногоИнтерфейсаOData`), наружу открываем только нужные объекты.
- **HTTP-сервис на 1С — запись.** Небольшое расширение конфигурации с одним сервисом: `POST /done` принимает факт выполнения и аккуратно проводит изменения в 1С (запись напрямую через OData в табличные части — хрупко, не делаем).

Аутентификация: отдельный сервисный пользователь 1С с минимальными правами (роль удалённого доступа OData + права на нужные объекты), basic-auth поверх HTTPS, IP-allowlist на VPS сервиса, если партнёр умеет.

### Режим B (план Б): 1С-инициатор — публикация не нужна

Если партнёр не публикует базу: расширение в 1С с **регламентным заданием** (каждые 1–5 минут), которое само ходит к API нашего сервиса:

- `GET https://drive.van-mark.ru/api/1c/outbox` — забрать подтверждения выполнения;
- `POST https://drive.van-mark.ru/api/1c/orders` — отправить новые/изменённые заказы.

Требование к партнёру минимальное: разрешить серверной базе исходящие HTTPS-запросы и установить расширение. Сервис аутентифицирует 1С по API-ключу в заголовке.

В обоих режимах на стороне сервиса один и тот же модуль `src/integrations/onec/` — меняется только транспорт (мы ходим в 1С / 1С ходит к нам).

## 3. Маппинг данных

| 1С (УНФ) | Сервис | Примечание |
|---|---|---|
| Документ-основание (Заказ покупателя / Заказ-наряд — **уточнить, чем реально пользуются**) | Task (externalId1C = Ref_Key) | создаётся в статусе NEW с пометкой «из 1С» |
| Контрагент.Наименование | task.orgName | |
| Контактное лицо + телефон | task.contactName / contactPhone | |
| Адрес доставки (контактная информация) | task.address | адреса в 1С бывают грязные — поле остаётся редактируемым |
| Номер счёта/заказа | task.invoiceNumber | |
| Дата отгрузки/выполнения | task.scheduledDate | |
| Состав заказа (номенклатура) | task.title / equipment | строкой, без позиций |
| — | статус, события, фото | живут только в сервисе; в 1С уходит итог |

Ключ синхронизации — `Ref_Key` (GUID объекта 1С) в `task.externalId1C`. Повторная выгрузка того же объекта — upsert, не дубль. Подтверждение в 1С — идемпотентно: свой UUID операции, 1С игнорирует повторы.

Что именно писать в 1С при выполнении (вариант для обсуждения с тем, кто сопровождает 1С): комментарий/реквизит в документе-основании + смена состояния документа; либо отдельный документ «Выполнение задачи» — решим по факту аудита конфигурации.

## 4. Надёжность

- Очередь исходящих подтверждений в сервисе (таблица `OutboxEvent`): если 1С недоступна — копим и повторяем с экспоненциальной задержкой. Ничего не теряем.
- Все обмены журналируются (таблица `SyncLog`: направление, объект, результат, ошибка). Экран «Синхронизация с 1С» в админке: последний обмен, очередь, ошибки.
- Конфликт правок (задачу изменили и в 1С, и в сервисе): поля заказа из 1С считаются мастер-данными и перезаписывают локальные правки, операционные поля (статус, водитель, фото) — мастер в сервисе. Зафиксировать в коде явно.

## 6. Что уточнить внутри компании (до начала фазы 2)

1. Каким документом в УНФ сейчас оформляются доставки/выезды (Заказ покупателя? Заказ-наряд? Счёт?) — от этого зависит маппинг. Спросить Милену/бухгалтера, посмотреть живые документы.
2. Все ли задачи водителей имеют документ в 1С (похоже, нет — «забрать СДЭК» вряд ли оформляется) — поэтому ручное создание задач остаётся.
3. Кто будет принимать решения по доработке на стороне 1С: партнёр-хостер, франч на подряде или никто (тогда расширение пишем сами через Claude Code + тест на копии базы).

## 7. Критерии готовности фазы 2

Заказ, созданный в 1С, появляется в сервисе задачей в течение ≤5 минут; выполнение задачи видно в 1С в течение ≤5 минут; обрыв связи на сутки не теряет ни одного события (очередь догоняет); журнал синхронизации показывает 100% операций.
