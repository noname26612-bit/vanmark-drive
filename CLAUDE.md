# CLAUDE.md — vanmark-drive

Сервис планирования и учёта задач водителей компании VanMark (van-mark.ru, листогибочное оборудование, выездной ремонт). Диспетчер Милена ставит задачи, два водителя-мастера выполняют: Алексей Каширский (Лада Ларгус, преимущественно выездные ремонты) и Алексей Писарев (преимущественно доставки). Этот файл — постоянные правила для Claude Code.

## Роли в проекте

- **Артём — владелец продукта.** Отвечает за логику, архитектуру и приоритеты. НЕ программист: не предлагай ему «посмотреть код» — объясняй решения по-человечески и показывай результат в работающем интерфейсе.
- **Claude Code — инженер end-to-end.** Закрывает все технические вопросы сам: код, тесты, миграции, отладка, деплой. Вопросы Артёму — только про бизнес-логику и продуктовые решения, не про технику.

## Документы (читать перед работой)

- `docs/PRD.md` — что строим: роли, типы задач, поля, статусы, экраны. Источник правды по продукту.
- `docs/ARCHITECTURE.md` — как строим: стек, схема Prisma, статусная матрица, API, безопасность. Источник правды по технике.
- `docs/ROADMAP.md` — порядок этапов и критерии приёмки. Работаем строго по этапам.
- `docs/archive/` — отложенные наработки (интеграция с 1С). Не источник правды, в работу не берём без явного решения Артёма.

Любое противоречие между кодом и этими документами — это баг: либо исправь код, либо предложи Артёму обновить документ. Молча отклоняться нельзя.

## Скиллы (использовать каждый раз)

- **feature-dev** — при реализации любой фичи/этапа. Рабочий цикл: спека → план → код → тесты → security-check.
- **security-check** — после ЛЮБОГО изменения API, запросов к БД или прав. Обязателен перед завершением этапа.
- **ui-guidelines** — при любой работе над интерфейсом (цвета статусов, размеры, тексты).
- **deploy-release** — при каждом деплое на сервер.

## Жёсткие правила (нарушать нельзя)

1. **Изоляция задач.** Водитель видит и меняет только задачи, где `assigneeId == его id`. Личность — только из сессии, никогда из тела запроса. Чужая задача по прямому ID → 404. Каждый новый эндпоинт проходит security-check.
2. **Статусная матрица** живёт в `src/domain/task-status.ts` и меняется только после согласования с Артёмом. Переходы в обход матрицы запрещены везде, включая админские ручки.
3. **TaskEvent — журнал только на запись.** События не редактируются и не удаляются. Исправление ошибки = новое событие.
4. **Фото при завершении — по желанию** (решение Артёма, этап 11). Жёсткого гейта «DONE без фото» больше НЕТ; поле `TaskType.requiresPhoto` устарело (DEPRECATED) и гейтом не является. Требование акта (`requiresSignedDoc`) — тоже не запрет завершения, а мягкая отметка KPI (`UNSIGNED_DOCS`) + бонус за комплектность. Единственная серверная проверка при DONE — отметка факта оплаты для типа `ON_SITE` (получено / не получено с причиной).
5. **Секреты** только в `.env` (в репо — `.env.example`). Никаких ключей/паролей в коде и коммитах.
6. **Не добавлять зависимости и сервисы** сверх ARCHITECTURE.md без обоснования Артёму (никаких «давай добавим Redis/очередь/микросервис»).
7. Деструктивные операции с продакшен-БД (drop, truncate, ручные UPDATE) — только после явного подтверждения Артёма и бэкапа.
8. **Проект держим на латинском пути.** Папка репозитория и все родительские каталоги — только из латиницы/цифр/`-_`, без кириллицы и пробелов (плохой случай: `…/Документы/проект`). Кириллица в пути ломает нативные модули (argon2, движки Prisma), pnpm, Docker bind-mounts и часть тулчейна. На проде — то же (например, `/opt/vanmark-drive`).
9. **Офлайн-очередь: никаких детерминированных 5xx.** Обработчик действия из офлайн-очереди (обёрнутый `withIdempotency`) на валидном входе не должен ронять повторяемую 5xx — иначе действие вечно `retryable`, а `break` в `sync.ts` намертво блокирует ВСЮ досылку водителя. Результат мутации-эффекта нормализуй (void → `Prisma.JsonNull`, не `undefined` в required Json); бизнес-отказы держи в 4xx (станут «конфликтом», не тупиком). Урок инцидента 06.07.

## Конвенции

- TypeScript strict, без `any` (исключение — обоснованный комментарием unknown-парсинг).
- Языки: интерфейс, тексты ошибок, пуши — на русском; код, коммиты, имена — на английском.
- Route handlers тонкие; вся логика в `src/domain/` (см. ARCHITECTURE §3).
- Ответы API: `{ data }` | `{ error: { code, message } }`.
- Коммиты: conventional commits (`feat:`, `fix:`, `chore:`); один этап Roadmap = одна ветка.
- Перед стартом проекта проверить актуальные стабильные версии стека (Next.js, Prisma, Auth.js, Serwist, Tailwind) в вебе — не брать из памяти.
- Service worker обязан переживать деплой без залипания клиента: держи `controllerchange → reload` в регистрации И храни предыдущую версию app-кэша (fallback чанков). При любой правке `public/sw.js`, стратегии кэша или смене `BUILD_ID` проверяй build-skew — старый открытый клиент против нового билда (удалённые с сервера чанки `/_next/static` → 404 → белый экран у водителя в поле).

## Definition of Done (для каждого этапа Roadmap)

1. Работает сценарий этапа из ROADMAP (критерии приёмки выполнены).
2. `lint` и `typecheck` чистые; unit-тесты домена зелёные; e2e-тесты критических потоков зелёные.
3. security-check пройден (если менялись API/БД/права).
4. Прогнано на телефоне (эмуляция Android/Chrome DevTools минимум) для водительских экранов.
5. Короткое демо-описание для Артёма: что сделано, как посмотреть, что решить дальше.

## Команды

```bash

pnpm prisma migrate dev   # миграции локально
pnpm test                 # vitest
pnpm e2e                  # playwright
pnpm lint && pnpm typecheck
```

## Чего НЕ делать

- Не переусложнять: 3 пользователя, ~10–30 задач в день. Поллинг вместо WebSocket, локальные файлы вместо S3, монолит вместо сервисов — это осознанные решения.
- Не делать «красиво потом»: ui-guidelines применяются с первого экрана.
- Не начинать фазу 2 (карты, аналитика и прочий пост-MVP) до приёмки фазы 1 Артёмом.
- Не возвращать интеграцию с 1С (архив в `docs/archive/`) без явного продуктового решения Артёма.
